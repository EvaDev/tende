Overview

1Voucher — How the system works

From onboarding to value movement

A Web2-first fintech with an onchain compliance and settlement layer. Consumers see a simple balance; the protocol holds claims in a Vault backed by treasury tokens (e.g. TTZA, TTMW).

Consumer app

  • Passkey wallet
  • Spend / save balances
  • Pay, remit, redeem vouchers

Admin console

  • Countries & currencies
  • Treasury ops
  • Merchants & compliance

Merchant console

  • POS & products
  • Accept payments
  • Settle to fiat
Phase 1 pilot: ZA corridor live; additional corridors (e.g. MWK / TTMW) follow the same pattern.

Admin setup

Admin: countries, currencies & treasury tokens

1
Add country
Countries table links ISO code → fiat currency (e.g. ZA → ZAR, MW → MWK). Controls KYC tiers and product availability.
2
Add treasury currency row
Currencies page: type TREASURY, code TTZA/TTMW, fiat anchor (ZAR/MWK). Saves to DB only — no chain tx yet.
3
Deploy corridor token (explicit action)
Deploy button creates ERC-1967 proxy + initialize; registers stablecoins row; wires Vault.setCurrencyTreasuryToken(fiat, proxy).
4
Configure corridors & partners
Remittance corridors, payout partners, FX overrides — off-chain routing for cross-border exits.
Saving a currency never auto-deploys onchain. Deploy requires admin confirmation and deployer wallet gas.
Consumer onboarding

Consumer registration → live wallet

1
Passkey created on device — WebAuthn P-256 key, no seed phrase.
2
Backend resolves Safe signer — SafeWebAuthnSignerFactory → deterministic signer address.
3
Consumer.registerConsumer onchain — Deploys Safe proxy; stores country hash, name hash, KYC level 0.
4
Off-chain profile (best-effort) — idOS credential, ENS subdomain, Pimlico paymaster whitelist.
5
consumers row in Postgres — Links wallet, country, KYC, registration step for retries.

Result: consumer has a gasless-capable Safe address registered onchain. Balance is still zero until cash-in.

Merchant onboarding

Merchant registration → can receive payments

1
Wallet signs nonce — POST /api/merchants/register proves wallet ownership.
2
merchants row — Name, country, settlement type, verification PENDING.
3
Onchain whitelist (best-effort) — TreasuryToken.addToWhitelist; Vault.setTrustedCounterparty.
4
Admin verification — Admin approves KYB → merchant goes live.
Trusted merchants bypass consumer-to-consumer country matching — consumers can pay them without P2P restrictions.
Value model

What consumers actually hold

Consumers see one fungible balance (e.g. R500). They never hold TTZA/TTMW ERC-20 in their wallet.

Spend balance (ZAR claim)

  • Vault ledger entry — ERC-4626-style shares pinned 1:1
  • Backed by TTZA sitting in the Vault
  • Flat 1:1 — no yield to consumer

Treasury token (TTZA, TTMW…)

  • Protocol bank-cash token — permissioned ERC-20
  • Mint on fiat-in; swept on merchant settlement
  • Closed-loop, domestic-only transfers

Accounting: TT in vault ≈ sum of all ZAR claims + unallocated pool. USD save balance is separate (USDC reserve).

Value flows

Cash-in: how value enters the system

After registration, a consumer needs a funded Vault claim before they can pay anyone.

1
Off-chain fiat or voucher — Bank deposit reference or voucher number in deposit_references.
2
Mint treasury token → Vault — Backend mints TTZA/TTMW to Vault contract.
3
Credit consumer claim — Vault.adminCredit(wallet, amount, ZAR).
4
Audit trail — deposit_references + indexer chain events.
POC: Admin Treasury → dev-credit simulates this on Sepolia. Production gates mint on reconciled bank deposit.

Technology

Onchain architecture

Consumer.sol

  • Safe wallet factory
  • Identity + KYC registry
  • Compliance event log

Vault.sol

  • Unified balance ledger
  • Remittance + FX paths
  • ERC-4626 reserve semantics

TreasuryToken.sol

  • Shared logic (one impl)
  • Compliance in _update hook
  • Mint/burn/freeze/whitelist

Each corridor (TTZA, TTMW…) is a separate ERC-1967 proxy instance — same logic, different initialize(name, symbol).

Summary

End-to-end: setup → funded → payment

1
Admin enables ZA + TTZA — Country, currency, deploy proxy, wire Vault.
2
Merchant registers + admin verifies — Whitelisted on TT + Vault trusted.
3
Consumer registers — Safe deployed, KYC level 0, zero balance.
4
Cash-in — Fiat/voucher → mint TT to Vault → credit ZAR claim.
5
Consumer pays merchant — Signed Vault.transfer; merchant ZAR claim increases.
6
Merchant settles (optional) — Claim debited, TT swept; bank payout off-chain.
Smart contracts = compliance ledger + settlement. Backend = orchestration, KYC, partners, and UX.
